# E-PoH Threat Model

## Trust Model

### Trusted Entities

| Entity | Role | Trust Assumption |
|--------|------|------------------|
| **Issuers** (hospitals, labs, oracles) | Hold secret keys; identity proven via witness in ZK circuit | Honest about health status; keys are secure |
| **Contract Owner** | Adds authorized issuers on-chain (max 4 slots) | Correctly vets issuers; multi-sig recommended for production |
| **Midnight Network** | Ledger state, transaction ordering, proof verification | Standard blockchain trust assumptions |
| **Proof Server** | Generates ZK proofs from circuit transcripts | Runs locally (Docker); does not see private state beyond witness execution |

### Untrusted Entities

| Entity | Assumption |
|--------|------------|
| **Badge holder (user)** | May attempt to forge, replay, or extend badges |
| **Verifier (third party)** | May attempt to extract private health data |
| **Network observers** | Can see ledger state but NOT private witness data |

## Threat Analysis

### T1: Replay Attack — Reusing someone else's proof

**Attack**: Attacker obtains a valid proof generated by another user and submits it to mint a badge for themselves.

**Mitigation**: The circuit computes `subject_hash = persistentHash(subject_address)` and discloses it to the ledger. The subject hash is cryptographically bound to the subject's address. A different user would produce a different subject hash, and the on-chain state would not match.

**Residual risk**: None — cryptographically bound to the subject.

### T2: Fake Issuer — Forging claims with an unauthorized key

**Attack**: Attacker generates their own secret key, creates a fake health claim, and produces a valid ZK proof.

**Mitigation**: The circuit derives the issuer's public key from their secret key via `derivePublicKey(isk, round)` and checks it against all 4 registered issuer slots using arithmetic verification: `m0 + m1 + m2 + m3 > 0`. An unregistered issuer's derived public key won't match any slot, and the circuit will fail the assertion.

**Residual risk**: Compromised issuer key. Mitigation: the owner can register a replacement issuer in one of the 4 slots. Previously minted badges remain valid until they expire — acceptable given the ephemeral design.

### T3: Expired Badge Reuse

**Attack**: User attempts to use an expired badge for verification.

**Mitigation**: Dual enforcement:
1. **On-chain**: Badge state checked via `last_badge_expires_at` and `last_badge_state`
2. **Off-chain**: Verification endpoint compares expiry timestamp against current time

**Residual risk**: None for standard verification flow.

### T4: Witness-Value Disclosure — Leaking which issuer signed

**Attack**: An observer analyzes the ZK proof or circuit execution to determine which of the 4 registered issuers signed the claim.

**Mitigation**: The Compact contract uses arithmetic sums instead of short-circuit OR (`||`) for issuer verification. The expression `m0 + m1 + m2 + m3 > 0` evaluates all 4 comparisons unconditionally, preventing information leakage through conditional branches. The Compact compiler enforces this — it rejects code that could disclose witness values through branching.

**Residual risk**: None — the circuit design prevents this by construction.

### T5: Double Minting — Minting multiple badges from the same claim

**Attack**: User submits the same claim data multiple times to get duplicate badges.

**Mitigation**: The contract tracks a `badge_count` and `round` counter. Each `mintBadge` call increments both, so the ledger reflects the total number of badges minted. Since badges are soulbound and expire, duplicate minting is low-impact.

**Residual risk**: Low. Duplicate badges don't grant additional privileges.

### T6: Privacy Leakage — Extracting health data from ledger state

**Attack**: Observer analyzes on-chain ledger state to determine medical details.

**What is revealed on the ledger**:
- `last_badge_claim_type` (0, 1, or 2) — general category, public by design
- `last_badge_expires_at` — expiry timestamp, public by design
- `last_badge_subject_hash` — persistentHash of subject address (irreversible)
- `last_badge_issuer_hash` — derived public key of issuer (not the secret key)
- `last_badge_state` — EMPTY, ACTIVE, or REVOKED

**What is NOT revealed**: Subject's actual address, specific medical records, test results, vaccination details, issuer's secret key, which specific issuer slot matched.

**Residual risk**: If a subject's identifier is publicly known, an observer could correlate badge ownership. Mitigation: use a dedicated identifier for health badges.

### T7: Issuer Key Compromise

**Attack**: An issuer's secret key is stolen, allowing the attacker to sign arbitrary health claims.

**Mitigation**:
1. Owner can register a new issuer in an available slot
2. All future badges from the new key will be accepted
3. Existing badges expire naturally (ephemeral design limits damage window)
4. For production: issuers should use hardware-secured key storage

**Residual risk**: Badges minted between compromise and response remain valid until expiry. Acceptable given short badge lifetimes (hours to days). Note: the current contract does not support removing issuers once added (4-slot limit), so key rotation requires deploying a new contract for production.

### T8: Contract Owner Abuse

**Attack**: Contract owner adds a malicious issuer or deploys a compromised contract.

**Mitigation**:
- For local dev/MVP: single owner is acceptable
- For production: governance via multi-sig or DAO
- All `addIssuer` calls are visible in the ledger state
- The contract's `authority` field shows the current owner's public key

## Summary Matrix

| Threat | Severity | Likelihood | Mitigation | Residual Risk |
|--------|----------|------------|------------|---------------|
| T1: Replay | High | Medium | Subject hash binding | None |
| T2: Fake issuer | High | Low | In-circuit issuer verification | Key compromise |
| T3: Expired reuse | Medium | High | On-chain + off-chain expiry check | None |
| T4: Witness disclosure | Medium | Low | Arithmetic sum (Compact compiler enforced) | None |
| T5: Double mint | Low | Medium | Badge count tracking | Low |
| T6: Privacy leak | Medium | Medium | ZK proof + persistentHash design | Address correlation |
| T7: Key compromise | High | Low | Ephemeral expiry + re-registration | Short window |
| T8: Owner abuse | Medium | Low | Multi-sig (production) | Governance risk |

# E-PoH Threat Model

## Trust Model

### Trusted Entities

| Entity | Role | Trust Assumption |
|--------|------|------------------|
| **Issuers** (hospitals, labs, oracles) | Sign health claims with Ethereum wallets (secp256k1) | Honest about health status; keys are secure |
| **IssuerRegistry Owner** | Adds/removes authorized issuers on-chain | Correctly vets issuers; multi-sig recommended for production |
| **Midnight Network** | Block timestamp, transaction ordering | Standard blockchain trust assumptions |

### Untrusted Entities

| Entity | Assumption |
|--------|------------|
| **Badge holder (user)** | May attempt to forge, replay, or extend badges |
| **Verifier (third party)** | May attempt to extract private health data |
| **Network observers** | Can see on-chain transactions but NOT private claim data |

## Threat Analysis

### T1: Replay Attack — Reusing someone else's proof

**Attack**: Attacker obtains a valid proof generated by another user and submits it to mint a badge for themselves.

**Mitigation**: The circuit outputs `subject_hash = Poseidon2(subject_address)`. The `EPoHBadge.mintBadge()` function checks that `subject_hash == Poseidon2(msg.sender)`. Since the attacker's address differs from the original subject's, the hash won't match and the transaction reverts.

**Residual risk**: None — cryptographically bound to the subject's wallet.

### T2: Fake Issuer — Forging claims with an unauthorized key

**Attack**: Attacker generates their own secp256k1 keypair, signs a fake health claim, and produces a valid ZK proof.

**Mitigation**: The circuit outputs `issuer_pubkey_hash = Poseidon2(issuer_pubkey)`. The `EPoHBadge.mintBadge()` checks `issuerRegistry.isIssuer(issuer_pubkey_hash)`. Unauthorized issuers are not in the registry, so the transaction reverts.

**Residual risk**: Compromised issuer key. Mitigated by `removeIssuer()` to revoke compromised issuers. Previously minted badges from a compromised issuer remain valid until they expire naturally — this is acceptable given the ephemeral design.

### T3: Expired Badge Reuse

**Attack**: User attempts to use an expired badge for verification.

**Mitigation**: Dual enforcement:
1. **On-chain**: `isValid(tokenId)` checks `expiresAt > block.timestamp` — returns false after expiry
2. **Off-chain**: Verification page/QR flow calls `isValid()` before displaying result

**Residual risk**: None for on-chain verification. Off-chain verifiers who don't check `isValid()` could be fooled by showing a badge exists — but the standard verification flow always checks.

### T4: Front-Running — Observing and stealing a proof from the mempool

**Attack**: Attacker monitors pending transactions, extracts the proof, and submits it before the original user.

**Mitigation**: Subject binding (T1 mitigation). Even if the attacker front-runs, `Poseidon2(attacker_address) != subject_hash` in the proof, so the attacker's transaction reverts. The original user's transaction succeeds in the next block.

**Residual risk**: None — the proof is cryptographically bound to the subject.

### T5: Double Minting — Minting multiple badges from the same claim

**Attack**: User submits the same proof multiple times to get duplicate badges.

**Mitigation**: The contract can optionally track `mapping(bytes32 => bool) usedProofHashes` and reject duplicate proof hashes. Alternatively, since badges are soulbound and expire, duplicate minting is low-impact — the user just has redundant badges that expire at the same time.

**Residual risk**: Low. Duplicate badges don't grant additional privileges.

### T6: Privacy Leakage — Extracting health data from on-chain data

**Attack**: Observer analyzes on-chain badge data to determine medical details.

**What is revealed on-chain**:
- `claim_type` (0, 1, or 2) — the general category is public by design (verifiers need to know what was proven)
- `expires_at` — the expiry timestamp is public by design
- `subject_hash` — Poseidon2 hash of address (cannot reverse to identify the person unless their address is already known)
- `issuer_pubkey_hash` — reveals which registered issuer category, not the specific institution

**What is NOT revealed**: Subject's actual address (hashed), specific medical records, test results, vaccination details, issuer's full identity.

**Residual risk**: If a subject's Ethereum address is publicly linked to their identity (e.g., ENS name), an observer could correlate badge ownership. Mitigation: users should use a dedicated wallet for health badges.

### T7: Issuer Key Compromise

**Attack**: An issuer's private key is stolen, allowing the attacker to sign arbitrary health claims.

**Mitigation**:
1. `removeIssuer()` revokes the compromised key from the registry
2. All future badges from that key will be rejected
3. Existing badges expire naturally (ephemeral design limits damage window)
4. For production: issuers should use hardware wallets or HSMs

**Residual risk**: Badges minted between compromise and revocation remain valid until expiry. Acceptable given short badge lifetimes (hours to days).

### T8: Contract Upgrade / Admin Abuse

**Attack**: Registry owner adds a malicious issuer or removes legitimate ones.

**Mitigation**:
- For testnet/MVP: single owner is acceptable
- For production: governance via multi-sig (e.g., Gnosis Safe) or DAO
- Timelock on admin actions (optional)
- Event logs (`IssuerAdded`, `IssuerRemoved`) provide transparency

## Summary Matrix

| Threat | Severity | Likelihood | Mitigation | Residual Risk |
|--------|----------|------------|------------|---------------|
| T1: Replay | High | Medium | Subject hash binding | None |
| T2: Fake issuer | High | Low | IssuerRegistry | Key compromise |
| T3: Expired reuse | Medium | High | On-chain timestamp | None |
| T4: Front-running | Medium | Low | Subject binding | None |
| T5: Double mint | Low | Medium | Optional proof hash tracking | Low |
| T6: Privacy leak | Medium | Medium | ZK proof design | Address correlation |
| T7: Key compromise | High | Low | Revocation + ephemeral expiry | Short window |
| T8: Admin abuse | Medium | Low | Multi-sig (production) | Governance risk |
